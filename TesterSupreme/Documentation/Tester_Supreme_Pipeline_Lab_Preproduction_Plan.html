01Purpose and Success Definition

This plan establishes the production pipeline before Tester Supreme gameplay begins.

Tester Supreme 0.0 is a controlled pre-production phase. Its purpose is to prove that assets can be modeled in Blender using inches, exported consistently, imported into s&box at the correct scale and orientation, converted into reusable prefabs, and assembled into modular rooms, and used safely by the built-in player controller.

The governing rule Gameplay production does not begin until a new model can travel from Blender to a usable s&box prefab without repeated manual correction.

What the pipeline must prove

  • The official Citizen human reference can be used in Blender as a reliable visual scale guide.
  • A verified FBX export preset produces correct scale, orientation, pivots, normals and material slots.
  • Modular floors, walls, corners, doorways and ceilings connect without visible gaps or collision seams.
  • Shared s&box materials can be assigned and survive reimporting.
  • Prefab instances update when the source model is revised.
  • The built-in player controller can walk, jump, fall, use stairs, cross seams and interact with the test kit.
  • The workflow stays reproducible and low-friction as the number of modular assets grows well past the size of this test kit — see Section 16.

Outside this phase

  • Weapons, enemies, progression, scoring and narrative content.
  • Final visual polish, unique wear, complex decals or production-quality texture variation.
  • Multiplayer, networking, monetization or public release preparation.
  • Large-scale architecture production before scale and snapping have been validated.
  • Performance optimization, LODs and draw-distance culling.

04Version Control and Environment Baseline

Binary art assets and a fast-moving engine both need explicit rules, or the pipeline quietly drifts.

Source control for binary assets

Track .blend, .fbx and image/audio source files with Git LFS rather than plain git. Plain git stores every binary revision in full inside history, which gets slow and heavy fast once the modular kit grows past a handful of assets — exactly the growth this project is planning for.

*.blend filter=lfs diff=lfs merge=lfs -text *.fbx filter=lfs diff=lfs merge=lfs -text *.png filter=lfs diff=lfs merge=lfs -text *.psd filter=lfs diff=lfs merge=lfs -text *.wav filter=lfs diff=lfs merge=lfs -text
  • Git LFS installed and initialized before the first binary asset is committed.
  • .gitattributes (above) committed before the Foundation milestone closes.
  • Compiled or generated s&box output (bin/, obj/, .generated/, cache/) stays out of version control.

Environment pinning

s&box is under active development and Blender's FBX exporter behavior varies by version. A calibration that passes today can silently fail after either tool updates. Record the environment the calibration was locked against, and re-check it whenever either tool changes.

Why this matters A calibration cube that measured correctly last month is not proof it still measures correctly today. Treat any Blender or s&box update as a reason to recheck, not just a reason to keep working.

Recalibration trigger

  • Blender updated since the last recorded environment.
  • s&box updated since the last recorded environment.
  • Re-export and re-measure cal_cube_12in against the current environment.
  • Record the result in TEST_LOG.md even when nothing changed.

  1. Create a new Game project using the cleanest available empty-project option.
  2. Save the first scene as Assets/scenes/development/pipeline_lab.scene.
  3. Add basic neutral lighting and a temporary floor.
  4. Add the standard playable character/controller from the current sample setup.

Calibration export procedure

  1. Select one calibration object and place it in _EXPORT.
  2. Apply transforms only after the origin is correct.
  3. Inspect face orientation and remove accidental internal geometry.
  4. Export one binary FBX per calibration object.
  5. Import through ModelDoc and place the model at scene scale 1,1,1.
  6. Measure and compare against the s&box grid and playable character.
  7. Change the Blender export preset rather than scaling each placed asset.
  8. Repeat until re-exporting produces the same size and placement every time.
  9. Record the current Blender/s&box versions in FBX_EXPORT_SETTINGS.md alongside the locked preset.

11Calibration Approval Gate

  • The 12-inch cube measures correctly in s&box.
  • The 96-inch floor aligns with the selected engine grid.
  • The 120-inch wall feels correct beside the actual playable character.
  • The asymmetrical axis marker faces the expected direction and is not mirrored.
  • Every placed model remains at scale 1,1,1.
  • Two floor modules meet without gaps or overlap.
  • Two wall modules meet without gaps or overlap.
  • Re-exporting the source does not move existing instances.
  • Blender and s&box versions are recorded alongside the locked preset.
  • The resolved Forward/Up combination is recorded explicitly, not left as "auto".
Lock only after passing Save the verified export preset, update the documentation, and archive TS_SBOX_MODELING_TEMPLATE_v1.blend only after every calibration item passes.

Template versioning after v1 is locked

  1. Do not edit TS_SBOX_MODELING_TEMPLATE_v1.blend in place.
  2. Duplicate it to a new versioned file (_v2) and make changes there.
  3. Re-run this entire approval checklist against the new version.
  4. Re-export and re-verify at least one already-approved test-kit asset against the new template before adopting it project-wide.
  5. Record the version change and the reason for it in TEST_LOG.md.
A newer template is not automatically better Nothing built against v1 should be assumed compatible with v2 until Bay F's reimport test (Section 24) has been repeated against it.

Material efficiency rules

  • Interactive props should normally use one or two materials.

  • Use shared materials and consistent texel density instead of unique textures for every wall.

Consistency ruleAll modular architecture shares one texel density; hero/interactive props may increase it, never decrease it

Lightmap UV (UV2)

Every architecture module intended for static placement gets a second, non-overlapping UV channel reserved for lightmapping, authored at the same time as UV1 — even though lightmapping itself is not validated until later.

  • UV1 (material) present.
  • UV2 (lightmap, non-overlapping, 0–1 space) present on architecture.
  • UV2 intentionally skipped for movable/interactive props is noted, not silently absent.
Why now Retrofitting a second UV channel onto an already-approved modular kit later means re-touching every approved asset. Reserving it during the Modular Test Kit milestone costs little and avoids that rework.

14Import into ModelDoc and s&box

  1. Export each reusable test-kit asset as its own binary FBX.
  2. Place source files in the matching Game/Assets/models category.
  3. Create or update the corresponding .vmdl in ModelDoc.
  4. Assign the final s&box material rather than relying on Blender viewport materials.
  5. Place the model in a test scene at scale 1,1,1.
  6. Confirm size, orientation, origin, normals, smoothing and material slots.
  7. Record the current Blender/s&box versions if this is the first import since the last recorded baseline (Section 04).
  8. Record any required exception in IMPORT_CHECKLIST.md.

  • Correct up direction

Collision authoring convention (per shape)

ShapeAuthoring method
Flat floor / wall / ceilingNative BoxCollider sized to bounds
ColumnNative BoxCollider, or CapsuleCollider if the silhouette reads as round

16Automation and Scaling Strategy

The Pipeline Lab validates the process by hand once. Tester Supreme's long-term goal is a framework that stays reliable as the asset count grows well past the test kit — that requires reducing how much of the process depends on a human repeating it correctly every time.

Why this matters now, not later

  • Bay F's reimport test (Section 24) is correct but manual; it does not scale past a handful of assets without becoming a chore that gets skipped under time pressure.
  • Skipped regression checks are exactly how "reliable modular workflow" quietly degrades into "worked once."

Recommended automation, introduced incrementally

MilestoneAutomation to introduce
0.0.2 — Modular Test KitA Blender script/macro that batch-exports every object in _EXPORT to individual binary FBX files using the locked preset, instead of exporting one at a time by hand.
0.0.4 — Interaction and ReimportA lightweight regression log (script or spreadsheet) tracking which test-kit assets have been reimport-tested since their last source change.
0.0.5 — Production Kit StartExtend the naming/pivot/collision conventions into a Blender validation script that flags an object before export if its pivot, scale or collection placement doesn't match Sections 09/15.
0.1+Revisit whether a fuller automated import/validation pass is worth the investment, based on how much manual regression testing actually cost during 0.0.x.
Automation means removing repetitive risk, not tooling for its own sake If a milestone's manual checklist is still fast and reliable, don't automate it yet. Automate the steps that are easy to get subtly wrong at scale, first.

24Bay F — Reimport and Update Test

  1. Place several instances of the same wall prefab.
  2. Return to Blender and modify one obvious visual detail.
  3. Re-export the FBX using the verified preset.
  4. Refresh or recompile the imported model.
  5. Confirm that every prefab instance updates.
  6. Confirm that instance position, rotation and scale remain unchanged.
  7. Confirm that material assignments remain connected.
  8. Confirm that collision still matches the model.
  9. Confirm the lightmap UV2 channel (Section 13), if present, survived the reimport.
  10. Repeat once with a material-slot-safe change.
  11. Log the result in the regression log introduced in Section 16, even when it passes.
Critical production test A pipeline that works only for first import is not enough. Tester Supreme depends on being able to revise modular assets without rebuilding the scene.
